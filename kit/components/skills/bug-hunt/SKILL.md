# Bug Hunt

Дисциплина: **сначала тест, потом фикс**. Сначала фиксируем багу в виде failing test, потом чиним так чтобы тест стал зелёным.

## Алгоритм (строго)

1. **Воспроизвести баг.** Запустить код или существующий тест. Если бага не воспроизводится у тебя — НЕ начинать фикс.

2. **Если нет minimal repro — написать.** Самая короткая программа/команда которая показывает bug.

3. **Написать failing test.** Разместить в существующей test suite проекта в стиле проекта (определить framework: pytest/jest/vitest/go test/cargo test). Тест должен ловить именно эту багу и упасть на текущем коде.

4. **Сформировать гипотезу root cause.** Не симптом — корень. Если не уверен — 3 гипотезы, проверить каждую (логи / breakpoint / minimal repro).

5. **Зафиксить root cause.** Не патчить симптом. Если симптом и root cause расходятся — отметить в коммит-сообщении почему фикс именно тут.

6. **Прогнать тест.** Должен стать зелёным.

7. **Прогнать остальные тесты.** Не сломали ли что-то.

8. **Регрессионный тест остаётся в репе.** Не удалять после фикса.

## Output

```
✓ Bug fixed
  Repro: <команда>
  Root cause: <короткий разбор>
  Test added: tests/<path>::<name>
  Other tests: <N> passed
  Files changed: <список>
```

## Когда делегировать debugger субагенту

Если bug сложный (не воспроизводится за 5 минут, замешано несколько систем, race condition / async) — делегировать `debugger` субагенту (opus) который методично пройдёт по своему алгоритму repro → isolate → 3 гипотезы → проверка → root cause + регрессионный тест.

## Триггеры

Фразы: "fix bug X", "не работает", "почему", "падает", "Error:", "Traceback", баг-репорт от пользователя.
