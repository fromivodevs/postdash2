# Safe Refactor

## Алгоритм

1. **Проверить тесты на refactor'имый код.** Если их нет — написать **базовые** перед рефакторингом (happy path + 1-2 edge cases). Не идеальные, но достаточные чтобы зафиксировать текущее поведение.

2. **Сделать рефакторинг.** Без изменения поведения. Никаких "пока тут — заодно фикшу X" — это смешает refactor и feature change.

3. **Прогнать тесты.** Должны быть зелёные **без изменений в самих тестах**. Если тесты пришлось править — это уже не refactor, а behavior change, отдельная история.

4. **Коммит refactor отдельным коммитом** от feature changes. Префикс: `refactor: ...`. Сообщение фокусируется на структурном изменении.

## Output

```
✓ Refactor complete
  Tests added before refactor: <N>
  Tests after refactor: all green, unchanged
  Files renamed: <N>
  Files split: <N>
  Commit: refactor: <message>
```

## Когда делегировать refactor-planner

Если рефакторинг затрагивает >5 файлов или несколько систем — сначала вызвать `refactor-planner` (opus) который вернёт **план** с риском каждого шага. Пользователь утверждает план, потом приступаем.
